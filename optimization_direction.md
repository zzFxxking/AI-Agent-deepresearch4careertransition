# AI求职助手 — L3→L4 关键升级路径

> **目标**：将项目从 "Agent系统构建者（L3）" 升级为 "前沿技术推动者（L4）"
> **核心思路**：构建可验证的评估体系 + 消融实验，量化证明每个模块的价值增量
> **预估耗时**：1-2 周 | **额外资源**：0（无需GPU，仅需人力标注）

---

## 1. 当前瓶颈诊断

### 1.1 L3 vs L4 的判定鸿沟

| 维度 | L3 现状 | L4 要求 |
|------|---------|---------|
| 评估方式 | LLM 自评（quality_score 0-100），无 ground truth | 有可验证的评估集，指标可复现 |
| 效果证明 | "质量评分稳定达85+"（自己评自己） | 有基线对比，有消融实验，有量化提升 |
| 设计论证 | 模块存在但无价值增量数据 | 每个模块的贡献可被量化 |
| 叙事结构 | "我做了 X" | "我发现 A 是瓶颈，用 B 优化了它，提升了 C%" |

### 1.2 最致命的问题

**面试官一句话就能击穿你的系统**：

> "你这个 Orchestrator-Workers 架构，比直接一条 prompt 让 DeepSeek 生成一份求职报告，到底好了多少？你的质量评分也是 LLM 打的，你怎么知道它不是在自己夸自己？"

你接了不这个问题，整个 L3 的立论就崩塌了。这是从 L3 到 L4 的 **唯一瓶颈**。

---

## 2. 解决方案：构建可验证评估体系

### 2.1 核心思路

L4 的核心不是 "用更多技术"，而是 **"证明设计决策的正确性"**。我们需要：

1. **构造 30 道有标准答案的测试 query**
2. **定义多维评估指标**（不再只用 LLM 打分）
3. **跑消融实验**（逐一去掉模块，看指标变化）
4. **产出量化分析报告**（LaTeX 或 Markdown）

### 2.2 第一步：构造 30 道测试 Query

测试集设计原则：**混合可自动验证的客观题 + 人工评分的主观题**。

#### 客观题（20 道，可自动验证）

| 类型 | 示例 | 验证方式 |
|------|------|----------|
| **事实查询** | "Transformer 的 Multi-Head Attention 有几个权重矩阵？分别是什么？" | 标准答案比对（Q/K/V/O） |
| **版本查询** | "LangChain 的最新稳定版本号是多少？0.3.x 系列最后一个版本是？" | API 实时验证 |
| **GitHub 项目信息** | "列出 GitHub 上 star > 10K、语言为 Python 的 Agent 框架" | GitHub API 交叉验证 |
| **概念覆盖** | "RAG 系统的核心组件有哪些？请逐一列出" | 标准清单比对（检索/生成/向量库/重排序等） |
| **时效性查询** | "截至2026年，支持MCP协议的主流框架有哪些？" | 网页搜索 + 人工确认 |
| **数值验证** | "Qwen2.5 系列中，最小的可商用模型参数量是多少？" | 官方文档比对 |

#### 主观题（10 道，人工评分）

| 类型 | 示例 | 评分维度 |
|------|------|----------|
| **学习路径** | "数学专业研一，3个月转大模型应用开发，给出学习路线" | 合理性/可行性/时效性 |
| **项目推荐** | "想做Agent方向，推荐3个适合入门的开源项目" | 项目质量/匹配度/非toy |
| **面试题生成** | "生成30道针对字节跳动大模型岗的八股题" | 覆盖度/前沿性/准确性 |
| **极端场景** | "只会Python基础，1个月内速成大模型算法岗" | 诚实性（是否劝阻）/合理性 |

#### 答案构建流程

```
对每道客观题：
  1. 知识来源确认（论文/官方文档/API查询）
  2. 构造标准答案 JSON（正确字段 + 可接受范围）
  3. 定义 match 规则（精确匹配 / 包含匹配 / 值域匹配）

对每道主观题：
  1. 定义 1-5 分评分标准（rubric）
  2. 3 人独立评分取平均（互评或朋友帮评）
  3. 计算评分者间一致性（Krippendorff's alpha）
```

### 2.3 第二步：定义多维评估指标

**告别单一 LLM 打分**，改用以下 6 个指标：

| 指标 | 计算方式 | 适用题型 |
|------|----------|----------|
| **事实准确率（Factual Accuracy）** | 生成答案与标准答案的一致比例 | 客观题 |
| **幻觉率（Hallucination Rate）** | GitHub 项目 star/语言/状态可验证为假的比例 | GitHub 相关题 |
| **覆盖召回率（Coverage Recall）** | 应覆盖的知识点中实际覆盖的比例 | 概念覆盖题 |
| **时效准确率（Freshness Accuracy）** | 版本号/日期相关答案的准确比例 | 版本查询题 |
| **人工通过率（Human Pass Rate）** | 人工评分 >= 3/5 的比例 | 主观题 |
| **综合得分（Composite Score）** | 上述指标的加权平均 | 所有题 |

### 2.4 第三步：消融实验设计

这是 L4 升级的**核心产出**。用同一套测试集，跑 5 组对比：

| 配置 | Orchestrator | Worker 数 | Critic | 迭代 | 预期定位 |
|------|:---:|:---:|:---:|:---:|------|
| **A（完整系统）** | ✓ | 8 | ✓ | ≤5 | 你当前系统 |
| **B（去 Critic）** | ✓ | 8 | ✗ | ≤5 | 去第二门控 |
| **C（去迭代）** | ✓ | 8 | ✗ | 1 | 单轮生成 |
| **D（去并行）** | ✓ | 1 | ✗ | 1 | 串行单 Worker |
| **E（纯 Prompt）** | ✗ | 0 | ✗ | 1 | 单次 LLM 调用 |

每个配置跑全部 30 道题，记录 6 项指标。**实验预期发现**（假设）：

```
已有假设（待验证）：
  A > E 在覆盖召回率上（多 Worker = 更多信息源）
  A < E 在幻觉率上（多轮迭代 = 多轮暴露给 LLM 幻觉累积）
  B < A 在覆盖召回率上（无 Critic = 更多盲区未能发现）
  C < A 在覆盖召回率和综合得分上（单轮 = 信息不够）
  D ≈ C（单 Worker 覆盖能力弱于8 Worker，但幻觉率更低）

关键发现可能是：
  配置 A-E 在事实准确率上差异不大（"
所有智能来自同一个 LLM" 的必然结果）
```

### 2.5 第四步：产出量化分析报告

```markdown
# 消融实验报告结构

1. 实验设置（测试集构成、评估指标定义、模型与参数）
2. 总体结果表（5 配置 × 6 指标矩阵）
3. 关键发现
   3.1 多 Agent 架构在覆盖度上提升 X%，以 Y% 的幻觉率增加为代价
   3.2 CriticWorker 的真实价值：拦住 Z% 的 toy 项目，提升面试题覆盖度 W%
   3.3 迭代优化的边际收益曲线（第 1 轮 → 第 N 轮的增量衰减）
   3.4 并行化的 scaling 曲线（1 Worker → 8 Worker 的效率变化）
4. 错误分析（每个配置的典型失败案例）
5. 架构改进建议（基于实验数据，而非直觉）
```

## 3. 为什么这条路径是最优选择

### 3.1 与 "微调 Critic" 路径的对比

| 维度 | 评估体系 + 消融实验 | 微调 Critic |
|------|-------------------|-------------|
| 资源门槛 | **0 额外成本**（无 GPU） | 需要 GPU + 大量标注数据 |
| 时间 | **1-2 周** | 2-3 周 |
| L4 说服力 | **高**（直接量化设计决策） | 中（用了技术上没证明必要性） |
| 面试可讲性 | "我量化证明了每个模块的贡献" | "我会调模型" |
| 可复用性 | **所有后续项目可用** | 限于此项目 |
| 技术风险 | **低**（不涉及训练） | 中（数据质量、训练稳定性） |

### 3.2 叙事升级对比

**L3 叙事（当前）**：
> 我设计了一个 Orchestrator-Workers 架构的 Agent 系统，包含双门控质量控制、OODA 搜索循环、MCP 工具集成。系统质量评分稳定达 85+。

**L4 叙事（完成后）**：
> 我构建了一套包含 30 道可验证测试题的评估框架，对 5 种系统配置进行了消融实验。实验发现：多 Agent 并行在信息覆盖度上提升 40%，但引入 8% 的额外幻觉；CriticWorker 能拦停 60% 的 toy 项目推荐；迭代超过 3 轮后边际收益降至 2% 以下。基于这些发现，我将第 3 轮后的 Critic 条件收紧，在保持覆盖度的同时将幻觉率降低至接近单轮水平。

### 3.3 之后再做微调 Critic

当你有了这套评估体系后，**微调 Critic 才有了意义**：

```
Before: "我微调了一个 Critic 模型"
After:  "我微调了一个 Critic 模型，在评估集上将 Critic-human 一致性
        从 LLM-Critic 的 Spearman 0.52 提升至 0.78，并在消融实验中验证
        了它比原 LLM-Critic 多拦截了 30% 的事实错误"
```

评估体系是地基，所有后续优化（微调、GRPO、RAG）都建立在地基之上。

## 6. FAQ

**Q: 为什么不做微调 Critic？不是说那才是后训练吗？**

A: 微调 Critic 需要训练数据，而训练数据需要 ground truth，ground truth 需要评估体系。评估体系是地基，微调是二楼。先打地基再盖楼。

**Q: 做完消融实验，发现某个模块几乎没用怎么办？**

A: 这反而是最有价值的发现。证明 "某个常见做法在你的场景下没用" 和证明 "某个方法有用" 同样是 L4 贡献。面试时讲出 "我原以为 OODA 循环很重要，但消融实验发现去掉它后指标基本不变，这说明多轮搜索的增量价值在搜索结果本身而不在循环结构"——这种洞察比 "我加了 OODA" 高明两个量级。

**Q: 30 道题够吗？**

A: 对证明 "我不是在自说自话" 而言，够了。GAIA benchmark 的验证集是 165 道题，SWE-bench Verified 是 500 道。你的目标不是发表 benchmark 论文，而是让面试官信服你的设计决策有数据支撑。
