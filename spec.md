# AI求职助手改造项目规格说明书 (spec.md)

> **版本**: v1.0  
> **日期**: 2026-05-11  
> **状态**: 需求已确认，待开发  
> **项目目标**: 将现有 Deep Research Agent 改造为面向 AI/大模型求职者的个性化学习路径与面试准备助手，用于暑期实习/秋招简历展示。

---

## 1. 背景与核心场景

### 1.1 开发者背景
- **身份**: 在校研究生，跨专业转型路径（数学 → 金融考研 → 计算机）
- **求职目标**: 2026年暑期实习 + 秋招
- **目标岗位**: 大模型应用/Agent开发（首选），算法工程师（备选）

### 1.2 核心痛点（真实经历驱动）
入门大模型阶段走了弯路：
- 过分关注八股背诵和刷 LeetCode，忽略项目实战的重要性
- 找到的八股资料**不成体系、内容过时**，未与前沿 AI 技术（如 RAG/Agent/MCP）贴合
- 项目选择缺乏判断力，选了**大众化的、toy 级别的、过时的项目**（如 miniMind），无法体现竞争力

### 1.3 目标用户
- **主要群体**: 非计算机科班或计算机科班但想转 AI/大模型方向的同学
- **典型画像**: 有基础编程能力，但对 AI 领域学习路径迷茫，需要系统化的八股整理和高质量项目推荐

### 1.4 使用场景
用户输入个人背景 + 求职目标，工具输出一份个性化的**面试准备报告**，包含：
- 从当前水平到目标岗位的**学习路径**
- 按知识点类别组织的**最新八股清单**
- 使用前沿 AI 范式、有真实含金量的**开源项目推荐**
- 针对目标岗位的**模拟面试问答**

## 2. 原项目能力评估

### 2.1 项目现状
现有项目是一个基于 **Orchestrator-Workers 架构**的 Deep Research Agent（Python + FastAPI + DeepSeek + Bocha 搜索），已有任务拆解、并行 Worker 搜索、迭代优化、SSE 流式输出、质量评分等能力，当前评分约 5.6/10。

### 2.2 能力评估矩阵

| 能力模块 | 评估结论 | 处理方式 |
|---------|---------|---------|
| Orchestrator 任务拆解 | 完全适用，学习路径生成天然需要拆解 | **直接复用** |
| 多 Worker 并行搜索 | 八股/项目/技术栈搜索可并行 | **直接复用** |
| 迭代优化循环 | 学习路径需多轮验证覆盖度和难度匹配 | **直接复用** |
| SSE 流式输出 | 长耗时生成需要实时进度反馈 | **直接复用** |
| Worker 工厂模式 | 新增 Worker 类型可直接接入 | **直接复用** |
| 来源质量评估 | 面经/GitHub 需调整评分规则，框架可用 | **部分复用** |
| 通用搜索 (Bocha) | 不适合垂直领域搜索 | **改造为垂直搜索** |
| 输出格式 (Markdown) | 不适合前端渲染和结构化消费 | **增加结构化 JSON 输出** |
| Prompt 模板 | 为研究报告设计，需改为面试语境 | **全面重构** |
| VisualizationWorker | 生成图表 JSON 对学习路径无用 | **删除** |
| AnalysisWorker | 数据趋势分析对面经推荐无用 | **弱化/合并** |
| 复杂学术引用 | 对学习路径太冗长 | **简化** |

### 2.3 purpose.md 中未实现模块的优先级

| 模块 | 优先级 | 原因 |
|------|--------|------|
| AgentMemory (持久化记忆) | **高** | 个性化需要记住用户画像，否则每次从零开始 |
| Critic Agent | **高** | 检查项目难度匹配度、八股覆盖度，现有 check_quality 太泛化 |
| CheckpointManager | **中** | 长耗时生成支持中断恢复 |
| RAG 层 | **中** | 向量索引历史面经/项目库，实现相似用户推荐 |
| MCP 协议 | **高**（已实现） | `deep_research/mcp/` 模块，已接入 GitHub MCP Server，支持精准仓库搜索 |

## 3. 改造目标与边界

### 3.1 核心问题定义
> **解决 AI 求职者的信息不对称问题**：帮助用户从海量、过时、低质量的信息中，筛选出最适合其背景的最新学习路径、系统化八股和高质量开源项目。

### 3.2 保留的功能
- Orchestrator-Workers（自适应 4-8 Worker）广度生成 + ReAct-Reflection 深度扩展双模式架构
- 任务拆解与优先级排序
- 并行 Worker 调度（ThreadPoolExecutor）
- 迭代优化与收益递减检测
- SSE 流式进度输出
- FastAPI 服务层（`api.py` 保留但不扩展）

### 3.3 新增的功能
1. **垂直搜索客户端**：
   - `GitHubSearchClient`：通过 GitHub API 搜索 stars/forks/语言/最近更新
   - `InterviewSearchClient`：定向搜索牛客/力扣/知乎等面经来源
   - 保留 `BochaSearchClient` 用于技术概念解释和官方文档搜索
2. **AgentMemory**：持久化用户画像（当前水平、薄弱点、已生成过的路径）
3. **Critic Agent**：审查项目难度匹配度、八股高频考点覆盖度
4. **结构化 JSON 输出**：供程序化处理和前端渲染
5. **面试模拟问答生成**：基于目标岗位和八股清单生成 Q&A

### 3.4 删除或替换的模块
- **删除**: `VisualizationWorker`
- **弱化/合并**: `AnalysisWorker` 合并到 `SearchWorker`
- **替换**: 复杂学术引用格式 → 简化来源标注（平台名即可）
- **替换**: 通用搜索 → 垂直领域搜索

### 3.5 绝对不做（边界）
- 学习进度追踪仪表盘（需要前端和用户系统，后续迭代）
- API 层的扩展和前端界面（保留现有 API 即可）
- 产品化（账号系统、多用户、付费等）
- 引入 Claude/GPT-4 等额外 LLM（仅使用手头已有 API）

## 4. 输入与输出定义

### 4.1 输入格式

#### 4.1.1 输入方式
**自然语言 + 可选结构化字段**（现阶段）。后续可扩展支持简历 PDF / GitHub 链接上传。

#### 4.1.2 结构化字段（三层递进模型）

**Tier 1 — 高决策价值（强烈建议填写）**：

| 字段 | 类型 | 说明 | 示例值 |
|------|------|------|--------|
| `target_role` | string | 目标岗位，直接决定八股范围和路径方向 | `"大模型应用开发"`, `"算法工程师"` |
| `time_budget` | string | 可用准备时间，决定路径密度 | `"3个月"`, `"6个月"` |
| `company_tier` | string | 目标公司层级，决定面试深度 | `"大厂"`, `"中厂"`, `"外企"` |

**Tier 2 — 中等决策价值（可选，填了更精准）**：

| 字段 | 类型 | 说明 | 示例值 |
|------|------|------|--------|
| `current_level` | string | 当前水平，决定路径起点 | `"零基础"`, `"有Python基础"`, `"做过1个小项目"` |
| `focus_areas` | list[string] | 希望重点加强的方向 | `["RAG", "Agent框架"]` |
| `avoid_areas` | list[string] | 明确不需要的方向 | `["前端", "嵌入式"]` |

**Tier 3 — 低决策价值（不作为结构化字段，由 Agent 自主推断）**：
- 当前最薄弱的方向（从技能列表和岗位 gap 分析中推断更准确）
- 目标技术栈偏好（由目标岗位反推最优技术栈，Agent 输出附带解释）
- 目标公司具体名单（维护成本高，用 `company_tier` + `target_role` 覆盖）

#### 4.1.3 最简输入示例

```json
{
  "query": "我是数学专业研一学生，跨考计算机，会Python和PyTorch基础，想找大模型应用开发方向的暑期实习，有3个月准备时间",
  "target_role": "大模型应用开发工程师",
  "time_budget": "3个月",
  "company_tier": "大厂",
  "current_level": "有Python和PyTorch基础，没做过完整项目",
  "focus_areas": ["RAG", "Agent框架", "MCP协议"],
  "avoid_areas": ["前端开发", "嵌入式"]
}
```

### 4.2 输出格式

#### 4.2.1 三轨输出
- **人类可读版本**：拆分为三份 Markdown 报告
  - `{task_id}_main.md`：主报告（执行摘要 + 学习路径）
  - `{task_id}_interview.md`：八股知识点清单 + 模拟面试问答
  - `{task_id}_projects.md`：项目推荐清单 + 每个项目的关联面试题
- **机器可读版本**: 结构化 JSON（`.json`），包含完整结构化内容，供前端/API 消费

#### 4.2.2 输出模块

1. **学习路径 (learning_path)**
   - 分阶段路线图（如"基础巩固期 → 项目实战期 → 面试冲刺期"）
   - 每阶段包含：时间分配、核心任务、验收标准
   - 根据 `time_budget` 调整密度和优先级

2. **八股清单 (interview_questions)**
   - **主维度**: 按"知识点类别"组织（如 Transformer / RLHF / RAG / Agent 架构 / MCP 等）
   - **灵活维度**: Agent 根据用户背景自主决定是否按"面试环节"（一面/二面/三面）或"目标公司"进一步分组
   - 每条包含：问题、参考答案要点、来源平台、出现频率标签

3. **项目推荐 (project_recommendations)**
   - 每个项目包含：名称、GitHub链接、stars/forks、技术栈、为什么适合该用户、建议贡献/学习路径
   - 按匹配度排序，过滤掉 toy 项目（要求使用较新 AI 范式、有真实应用场景）

4. **面试模拟问答 (mock_interview)**
   - 基于目标岗位和八股清单生成的 Q&A 对
   - 包含：问题、考察点、参考答案、追问方向

## 5. 非功能性需求

### 5.1 模型与搜索策略
- **LLM**: 仅使用手头已有 API（DeepSeek-V4-pro / Kimi-k2.6 / 千问），不引入额外模型
- **搜索源**:
  - Bocha：技术概念解释、官方文档
  - GitHub API：开源项目搜索（stars/forks/语言/更新时间过滤）
  - 面经站点（牛客/力扣/知乎）：定向面经搜索

### 5.2 性能要求
- **耗时策略**: 接受用时间换质量，单次生成允许 2-5 分钟
- **熔断机制**: 必须设置超时熔断（如最大 20 分钟），防止无限迭代
- **并发**: Worker 并行搜索控制线程池大小，避免 API 限流

### 5.3 隐私与成本
- 数据存储：用户画像和学习历史本地持久化（SQLite 或 JSON 文件），不上传云端
- API 费用：接受个人 API Key 产生的合理费用，面经/GitHub 搜索结果本地缓存减少重复调用

### 5.4 易用性
- **主要交互**: 命令行（`python -m deep_research` 或脚本调用）
- **辅助交互**: 保留现有 FastAPI 服务用于本地演示，但不扩展新端点
- **部署**: 提供 Dockerfile 或一键运行脚本，确保面试官可快速复现

### 5.5 代码质量
- 新增模块需包含基础单元测试（尤其是 Critic Agent 和搜索客户端）
- 保留类型注解和 Pydantic 模型验证
- 关键决策点（如项目筛选规则、难度匹配逻辑）需有注释说明

## 7. 关键决策记录
